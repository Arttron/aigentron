# Поэтапный план реализации

> Превращает [`recommendations.md`](./recommendations.md) в упорядоченную дорожную карту с учётом
> зависимостей. Опирается на текущую реализацию, описанную в
> [`implementation-local-server.md`](./implementation-local-server.md). Работа по каналам и
> людям-агентам из [`rfc-001-channels-and-human-agents.md`](./rfc-001-channels-and-human-agents.md)
> встраивается там, где указано (она использует тот же краеугольный камень — идентичность из Фазы 1).

> Каждая фаза: **Цель · Почему сейчас · Объём (с привязкой к файлам) · Модель данных · Зависит от ·
> Готово когда · Риски.** Фазы последовательны там, где стрелка (→) показывает жёсткую зависимость;
> независимые фазы могут идти параллельно.

## Фаза 0 — Укрепление ops и предполётные проверки — ✅ ГОТОВО (2026-07-02)
**§8. Независима, дёшево, делать первой — чтобы перестать терять время на известные грабли.**

незаметно, и добавить единственный smoke-тест, который ловит самую частую поломку локальной модели

## Фаза 1 — Модель идентичности и ролей  ★ краеугольный камень — ✅ ГОТОВО (2026-07-02)
**§2. Предпосылка для каналов, ревью человеком и федерации.**

> **Статус: реализовано и проверено.** Миграция `20260702120000_identity_model` применена;
> оба пакета проходят typecheck; сценарии проверены на живом API (seed operator; атрибуция
> задач; guard: reviewer→403 на создании задачи, task_setter→403 на approval-decision,
> неизвестный актор→403; identities в 2 каналах). **Отклонения от плана (осознанные):**
> - `ApprovalRequest.resolvedBy` (строка) **сохранён** рядом с новым FK `resolvedById` —
>   недеструктивно и оставляет метки вроде `system:timeout`, у которых нет `User`. Аналогично
>   `Task` получил `createdById` (новое поле), а не переименование.
> - Идентификация актора — заголовок **`x-lds-user`** (id пользователя); отсутствует → default
>   operator. Guard применён к `POST /tasks` и `POST /approvals/:id/decision`; управление
>   пользователями (`/api/users` POST/PUT/DELETE) — `operator|admin`. Остальные management-роуты
>   (providers/agents/settings/mcp) пока НЕ загейчены (follow-up), чтобы не ломать дашборд.
> - Тестов jest в репо нет → проверка сделана end-to-end через API, а не unit-спеком.

**Цель:** система знает, *кто* совершил каждое действие. Ввести минимального `User` с ролями и
идентификаторами в каналах, привязать создание задач и аппрувы к нему — не строя полноценный auth.

**Объём:**
- Новая сущность `User` и `ChannelIdentity` (один человек → много id в каналах). Роли:
  `operator | reviewer | task_setter | admin`.
- Привязать `Task.createdBy` и `ApprovalRequest.resolvedBy` к `User.id` (сегодня `resolvedBy` —
  свободная строка; мигрировать её). Обновить сигнатуры + DTO `tasks.service.create` и
  `approvals.service.decide` (`@lds/shared` `ApprovalDecisionInput.decidedBy`).
- **Ролевое ограничение на уровне API** (не только UI): NestJS-guard, мапящий вызывающего → `User`
  → разрешённые действия. `task_setter`: создание/комментарии. `reviewer`: approve/reject.
  `operator`/`admin`: дополнительно подтверждения merge/push/deploy + управление
  провайдерами/агентами/настройками. Для локального развёртывания с одним оператором резолвинг
  «вызывающий → пользователь» может начинаться с единственного засеянного оператора; сейчас
  обязательны именно *модель данных и guard*.
- Дашборд: показывать действующего пользователя; минимальный менеджер пользователей (позже общий с
  админкой каналов).

**Модель данных (Prisma):**
```prisma
model User {
  id          String   @id @default(cuid())
  displayName String
  role        String   // operator | reviewer | task_setter | admin
  createdAt   DateTime @default(now())
  identities  ChannelIdentity[]
  tasks       Task[]            @relation("TaskCreator")
  approvals   ApprovalRequest[] @relation("ApprovalResolver")
}
model ChannelIdentity {
  id        String @id @default(cuid())
  userId    String
  user      User   @relation(fields: [userId], references: [id], onDelete: Cascade)
  channel   String // 'dashboard' | 'slack' | 'telegram' | 'email'
  externalId String
  @@unique([channel, externalId])
}
// Task.createdBy → User?, ApprovalRequest.resolvedBy → User? (FK, было String)
```

**Зависит от:** ничего (но всё, обращённое к людям, зависит от *неё*).

**Готово когда:** каждая новая задача и каждое решение по аппруву несут реальный `User.id`;
ролевой guard отклоняет недостаточно привилегированного вызывающего на уровне API (проверено
тестом); `ChannelIdentity` может связать одного пользователя в ≥2 каналах.

**Риски:** миграция существующих строк `resolvedBy` (бэкфилл на засеянного оператора). Расползание
в полноценный auth — сопротивляться; SSO/пароли явно вне scope v1.

**Связь с RFC-001:** эта модель `User`/`ChannelIdentity` *и есть* основа `ChannelUser` из RFC —
строим её здесь один раз. Каналы (входящее управление, человек-как-агент) можно начинать сразу
после этой фазы.

## Фаза 2 — Устойчивая общая шина событий  ★ краеугольный камень — ✅ ГОТОВО (2026-07-02)
**§4. Необходимая инфраструктура до масштабирования или федерации.**

> **Статус: реализовано и проверено.** `AgentEventBus` переведён с in-process `EventEmitter`
> на **Redis Pub/Sub** (канал `lds:bus`), интерфейс `publish`/`subscribe` не изменился. Локальный
> `EventEmitter` оставлен как in-process fan-out; входящий транспорт — отдельное subscribe-соединение
> (`redis.duplicate()`), ioredis авто-переподписывается после разрыва. Typecheck чистый; при старте
> лог `Subscribed to Redis channel "lds:bus"`; проверено, что событие `task-upserted` реально уходит
> в канал (redis-cli SUBSCRIBE поймал полный payload). Postgres остаётся источником истины —
> транспорт fire-and-forget, без replay.
>
> **Решение по транспорту (точка §4):** выбран **Pub/Sub**, а не Streams. Обоснование: федерация —
> это Фазы 7–8 (не ближняя цель), критерии приёмки Фазы 2 (мультиинстанс-доставка + long-poll)
> Pub/Sub закрывает, а долговечность транскрипта уже даёт Postgres. Интерфейс шины сохранён
> байт-в-байт, поэтому переход на **Redis Streams** (consumer-groups/replay) под Фазу 8 —
> локальная замена реализации `AgentEventBus`, без правок продюсеров/консюмеров.
>
> **Не делалось (осознанно):** второй экземпляр оркестратора физически не поднимался — доказан
> путь публикации в общий канал + подтверждена подписка процесса; мультиинстанс следует из дизайна
> (все инстансы читают один канал). Реальный прогон двух инстансов — при работе над Фазой 8.

**Цель:** заменить однопроцессную in-memory `AgentEventBus` на транспорт поверх Redis, чтобы второй
экземпляр оркестратора (а позже — удалённый узел) мог наблюдать тот же поток событий. Postgres
остаётся устойчивым источником истины для строк `AgentEvent`.

**Объём:**
- Сохранить интерфейс `AgentEventBus` (`publish`/`subscribe`) — поменять реализацию с Node
  `EventEmitter` на Redis (Redis уже в стеке ради BullMQ). Производители/потребители
  (`events.gateway.ts`, `approvals.service.waitForVerdict`, executor, tasks) не трогаются за
  интерфейсом.
- **Выбор транспорта — точка решения:**
  - **Redis Streams (рекомендуется)** — consumer groups + replay-from-offset; естественная основа
    для второго экземпляра и для федерации (Фазы 7/8). Выбрать это, *поскольку федерация из §6 в
    дорожной карте.*
  - **Redis Pub/Sub** — меньше изменений, fire-and-forget, без replay. Приемлемо, только если от
    федерации отказываемся.
- Socket.IO остаётся слоем, обращённым к браузеру; он подписывается на Redis, а не заменяется им.
- `approvals.waitForVerdict` сейчас подписан на in-process шину — перенаправить на подписку Redis;
  сохранить немедленную перепроверку БД, покрывающую гонку «вердикт пришёл первым».

**Модель данных:** нет (ключи Streams — рантайм, не схема). Опционально короткий TTL/maxlen у
Stream.

**Зависит от:** ничего жёсткого; лучше сделать рано, так как Фазы 7/8 её требуют.

**Готово когда:** два экземпляра оркестратора на одном Redis оба стримят живые логи задачи своим
клиентам дашборда; рестарт одного экземпляра не роняет поток другого; long-poll аппрувов всё ещё
резолвится.

**Риски:** семантика порядка/at-least-once у Streams (потребители должны быть идемпотентны — gateway
уже ключует строки UI по `sessionId:seq`, дедуп на месте). Потеря сообщений Pub/Sub при коротких
разрывах — почему Streams предпочтительнее.

## Фаза 3 — Failover провайдера/модели + политика ретраев  ★ краеугольный камень — ✅ ГОТОВО (2026-07-02)
**§3. Защищает гибридную экономику local/cloud.**

> **Статус: реализовано и проверено на живом прогоне.** Без миграции/изменений shared —
> только оркестратор + дашборд (typecheck обоих чистый).
> - **Цепочка fallback на агента:** frontmatter `fallbackProviders` (csv), проброшено через
>   agent-registry (parse/serialize) + DTO/контроллер + форму агента в дашборде
>   (CheckboxGroup провайдеров).
> - **Исполнитель:** `run()` разбит на provider-independent setup + цикл по цепочке
>   `[primary, ...fallbacks]` (дедуп, пропуск отсутствующих/без-модели) → `attempt()` на один
>   провайдер (своя `AgentSession`). При provider-level сбое — переход к следующему; в
>   транскрипт пишется строка `↻ failing over from "X" (reason) → "Y"`. Резюм follow-up — только
>   на первой попытке.
> - **Классификация:** throw из `runAgent` (не-abort) = транспортный/фатальный сбой провайдера
>   → **failover всегда**; `result.isError` (модель ответила ошибкой) → failover только по
>   паттернам `isFailoverWorthy` (conn/4xx/5xx/rate-limit/auth/model/tool). Abort/timeout → НЕ
>   failover (это стоп оркестратора → `stalled` из Фазы 4). Границей ретраев служит сама цепочка
>   (каждый провайдер — одна попытка); `attempts:1` в BullMQ сохранён.
> - **Проверено E2E:** агент с `provider=broken` (битый эндпоинт) + `fallbackProviders=claude-cloud`
>   → сессия `broken` errored → **failover** → сессия `claude-cloud` completed/reported `done` →
>   задача `done`; строка failover есть в транскрипте.
>
> **Отклонения / отложено (осознанно):**
> - **Enforce smoke-теста structured-tool-use (отказ маршрутизировать на модель без tool-use)** —
>   зависит от Фазы 0 (не сделана). Отложено вместе с M0.
> - **Failover по «no-progress таймауту»** — не реализован: через LiteLLM (`request_timeout: 600`)
>   быстрый провайдерский сбой воспроизвести трудно, а обычный timeout осознанно НЕ триггерит
>   failover (→ `stalled`). Ретрай того же провайдера на транзиентной ошибке — тоже follow-up
>   (сейчас только переход по цепочке).

**Цель:** транзиентный сбой или отказ локальной модели больше не убивает задачу сразу — executor
идёт по явной цепочке fallback и ретраит по-настоящему повторяемые сбои ограниченное число раз,
уважая неидемпотентность запусков.

**Объём:**
- **Цепочка fallback** на агента во frontmatter (напр. `fallbacks: claude-cloud, coder`). Executor
  (`real-agent-executor.resolveModelEnv` / `run`) идёт по ней при: недоступности/ошибке auth
  провайдера, некорректном/неструктурированном tool-call от локальной модели, либо таймауте «нет
  прогресса» ниже порога.
- **Классификация повторяемых и неповторяемых** сбоев: ретраить только сетевые сбои / rate limit,
  до небольшого ограниченного числа; никогда не авто-ретраить сбои плохого промпта / неправильного
  использования инструментов. Это остаётся на уровне executor, *не* BullMQ `attempts` (запуски
  остаются неидемпотентными; держим `attempts: 1`).
- **Принудить** smoke-тест structured tool-use из Фазы 0: отказываться маршрутизировать трафик
  агента на модель, не прошедшую проверку структурированного вывода, вместо обнаружения посреди
  задачи.
- Показывать failover в транскрипте как событие `agent:log` вида `provider_failover`, чтобы
  ревьюер видел, почему модель сменилась посреди прогона.

**Модель данных:** `AgentSession` может фиксировать фактически использованные провайдера/модель
(уже есть `provider`/`model`) — добавить строку на каждую попытку fallback либо индекс `attempt`,
чтобы транскрипт и подсчёт стоимости (Фаза 5) атрибутировали корректно.

**Зависит от:** Фаза 2 рекомендуется (события failover по устойчивой шине); smoke-тест из Фазы 0.

**Готово когда:** убийство основного провайдера посреди задачи переводит прогон на следующий в
цепочке с видимой строкой `provider_failover`, и задача всё же завершается; модель, не прошедшая
проверку tool-use, отклоняется на этапе маршрутизации.

**Риски:** отличить «таймаут без прогресса» от «легитимно медленной локальной модели» — сделать
порог конфигурируемым и консервативным; ложный failover жжёт токены.

## Фаза 4 — Контракт структурированного завершения задачи + watchdog  ★ краеугольный камень — ✅ ГОТОВО (2026-07-02)
**§1. Убирает зависимость от того, что модель «вспомнит сказать, что закончила».**

> **Статус: реализовано и проверено на живом прогоне (claude-cloud/Haiku).** Оба пакета
> typecheck; миграция `20260702140000_status_contract` применена (enum `blocked`/`stalled`,
> `AgentSession.reportedStatus/reportedSummary/lastHeartbeatAt`).
> - **`report_task_status` + `heartbeat`** — как in-process SDK MCP-сервер (`lds`) в
>   `@lds/agent-runner` (zod), хендлеры зовут колбэки исполнителя. Терминальный статус задачи
>   решает **worker по outcome**, а не по прозе: `blocked`/`failed` из отчёта — как есть;
>   аномальный конец без отчёта `done` → **`stalled`**; строгий режим (`REQUIRE_STATUS_REPORT`,
>   по умолчанию off) — чистый конец без отчёта → `stalled`.
> - **`AgentRunOutcome`** возвращается из `executor.run()` (перестал бросать на agent-error);
>   отчёт/heartbeat персистятся на сессию.
> - **Проверено E2E:** агент вызвал `report_task_status(done)` → задача `done` **без запроса
>   аппрува**; `AgentSession.reportedStatus='done'` записан.
> - **Найден и починен баг:** классификатор помечал ВСЕ `mcp__*` как опасные, из-за чего
>   `report_task_status` уходил в `needs_approval`. Добавлено исключение `mcp__lds__*`
>   (внутренние control-plane инструменты) в `@lds/shared` classify — auto-allow.
>
> **Отклонения / отложено (осознанно):**
> - **Watchdog «re-prompt один раз, затем stalled» — упрощён.** Стриминговый SDK не даёт чисто
>   инъектировать промпт в идущий `query()`, поэтому «зависший» ран ловится существующим
>   **hard-timeout** (`AGENT_RUN_TIMEOUT_MS`) → abort → `stalled`. `heartbeat` пишет
>   `lastHeartbeatAt` (видимость liveness). Полноценный no-progress watchdog + однократный
>   re-prompt — follow-up.
> - `stalled`/`blocked` маппинг реализован и типизирован в одном switch финализации; live
>   прогонялся happy-path (`done`) — ветки `stalled`/`blocked` не гонялись вживую (симметричны).

**Цель:** задача достигает терминального состояния только по явному сигналу, а прогон, завершившийся
без него, автоматически обнаруживается и восстанавливается — а не остаётся в неопределённости.

**Объём:**
- Обязательный инструмент `report_task_status(status, summary, files, handoff)` в схеме каждого
  агента (MCP- или SDK-инструмент). Строка в Postgres переходит в `done | failed | blocked`
  **только** при вызове этого инструмента — никогда из парсинга прозы `result`/`stderr`.
- **`stalled`, принуждаемый рантаймом:** если прогон завершается по *любой* причине (max turns,
  таймаут, краш, дисконнект) без вызова инструмента, **обработчик завершения прогона оркестратора**
  (`real-agent-executor.run` finally-block / worker) помечает задачу `stalled` и эмитит событие
  `task:stalled`. Это логика оркестратора, а не инструкция в промпте.
- **Watchdog-таймер** на задачу (конфигурируемый, по умолчанию пропорционально
  агенту/сложности): по истечении без отчёта о статусе или heartbeat — авто-переспросить агента о
  статусе один раз, затем `stalled`.
- Опциональный лёгкий инструмент `heartbeat(progress)` для долгих задач, чтобы оркестратор отличал
  «ещё работаю» от «молча застрял».

**Модель данных:** добавить `stalled` (и `blocked`) в `TaskStatus`; `lastHeartbeatAt` на `Task`
или `AgentSession`.

**Зависит от:** Фаза 2 (`task:stalled` по устойчивой шине). Независима от Фазы 3, но обе — работа
по надёжности прогонов и могут войти в одну веху.

**Готово когда:** задача, убитая посреди прогона (без инструмента статуса), автоматически попадает
в `stalled` с событием `task:stalled`; нормальный прогон терминализуется только через
`report_task_status`; watchdog переспрашивает один раз перед тем, как сдаться.

**Риски:** путь `stalled` должен быть идемпотентен с отменой (переиспользовать атомарный guard
`updateMany` из `resumeTaskIfClear`). Переспрос watchdog не должен гоняться с ещё выполняющимся
прогоном.

самопохвале — задокументированный failure mode). Держать её отдельным запланированным процессом.

## Фаза 8 — Федерация, Уровни 2/3: бэкенд исполнения и передача задач
**§6 Уровни 2–3. Самый высокорисковый; браться только после стабилизации Фаз 1, 2, 3.**

**Цель:** исполнять tool/shell задачи на удалённом хосте (L2), затем передавать целые задачи
удалённым узлам (L3) со строгими границами доверия.

**Объём:**
- **L2 — удалённый бэкенд исполнения:** shell/tool-исполнение задачи идёт по SSH на удалённом
  хосте, а состояние/история/навыки остаются на исходном оркестраторе.
- **L3 — полная федерация задач:**
  - **Реестр флота** поверх Redis Streams: heartbeat `{nodeId, capacityFree, modelsAvailable,
    projectScopes}`, чтобы роутер мог спросить «кто может это взять».
  - **Git-как-транспорт** для передачи (клон ветки → удалённый работает → пушит feature-ветку →
    возвращает ссылку на PR) вместо синхронизации рабочих директорий.
  - Перенять форму контракта task/artifact/status из **A2A (Agent2Agent)** Google, а не изобретать
    свой — даже частичное принятие облегчит будущую interop. Согласуется с контрактом
    `report_task_status` из Фазы 4.
- **Сквозные правила доверия (жёсткие требования):**
  - Флаг `delegatable` на задаче/типе, по умолчанию **false** — только явный opt-in.
  - **Никакие креды/секреты не путешествуют** с делегированной задачей; работа, требующая секрета,
    остаётся локальной.
  - Результаты с удалённого узла возвращаются как **предложение (diff/PR)**, никогда как прямой
    merge — тот же гейт merge/push/deploy применяется без изменений.
  - Считать удалённый узел скомпрометированным: никогда не слать больше, чем scoped-задачу.

**Модель данных:** `Task.delegatable Boolean @default(false)`; реестр флота (Redis, не схема);
аудит-строки узлов/передач по мере надобности.

**Зависит от:** Фаза 1 (идентичность), Фаза 2 (устойчивая/Streams-шина + реестр), Фаза 3 (failover).
L3 также опирается на контракт статуса из Фазы 4.

**Готово когда (L1→L2→L3, каждый независимо полезен):** L2 выгружает build/test на удалённый хост с
сохранением локального состояния; L3 передаёт `delegatable`-задачу удалённому узлу, который
возвращает ссылку на PR, без отправки секретов и с принуждённым merge-гейтом на результате.

**Риски:** граница доверия — это вся суть; не выпускать L3 без scoping кредов и правила
«предложение, не merge». Песочница на принимающем узле — ответственность принимающего оператора,
но отправитель обязан считать её враждебной.
